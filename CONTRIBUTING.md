# CONTRIBUTING.md — 共同編集ガイド（5人チーム）

> このプロジェクトは**5人が同じリポジトリを共同編集する前提**で運用します。
> 目的は「壊さず・止めず・矛盾なく」締切（**2026/6/22 11:59AM JST**）に間に合わせること。
> 制作仕様の正本は `outputs/README.md`、共通規約は `CLAUDE.md`。

---

## 1. 共同編集の3原則

1. **担当ファイルを跨いで勝手に直さない。** 気づいた他担当の問題は直接いじらず、`STATUS.md` の「気づき」か Slack に書く。
2. **数値変更は単独で完結させない。** `CLAUDE.md §3` の波及ルール（02→03→04→05）に従い、下流担当に連絡する。
3. **作業前後に `STATUS.md` を更新する。** 状態は人の頭ではなくファイルに置く。

---

## 2. ファイルオーナーシップ（誰がどれを編集するか）

衝突を避けるため、**1ファイル＝主担当1人**を原則とする。

| ファイル | 主担当 | 役割 |
|---------|--------|------|
| `outputs/README.md` ／ ファンド名・条件 ／ 最終提出 | **リード** | 全体統括・QA司令塔 |
| `outputs/01_前提パック.md` | **市場/競合担当** | 出典の一次確認・最新化 |
| `outputs/02_財務モデル.xlsx` | **財務担当** | 前提セルを詰めて妥当域を確定 |
| `outputs/03_スライド骨子.md` ／ `05_発表スライド.pptx/.pdf` | **資料担当** | 作図・デザイン・尺管理 |
| `outputs/04_プレゼン台本.md` | **発表担当** | 台本・読み合わせ・本番録画 |
| `STATUS.md` ／ `CLAUDE.md` ／ `CONTRIBUTING.md` | **リード**（更新は全員可） | 運用ドキュメント |

> 名前の割り当ては `STATUS.md` の「役割分担」表で管理する。

---

## 3. テキスト vs バイナリ（**競合対策・最重要**）

- **`.md` ファイル** … テキストなので Git で行単位マージ可。安心して並行編集してよい。
- **`.xlsx` / `.pptx` / `.pdf`** … バイナリ。**Git は中身をマージできない＝同時編集すると片方の作業が消える。**

### バイナリのロック運用（必須）
1. 編集する前に **Slack で「02触ります」と宣言**（簡易ロック）。
2. 編集したら**すぐ commit & push**して握りを短くする。
3. 終わったら **Slack で「02離しました」と宣言。**
4. 同じバイナリを2人同時に開かない。どうしても必要なら別名コピーで作業し、後で主担当が手で統合する。

> `.gitattributes` でバイナリは `merge=binary` 指定済み（自動マージ事故を防ぐ）。それでも**運用ロックが本命**。

---

## 4. Git 運用

軽量運用。学生課題なので過剰なフローは作らない。

- **ブランチ：** 各自 `feat/<担当>-<内容>`（例 `feat/finance-multiple`）。`main` への直 push は原則しない。
- **こまめに pull：** 作業開始前に必ず `git pull`。バイナリ編集前は特に。
- **コミット粒度：** 「1つの意味のある変更＝1コミット」。巨大コミットにしない。
- **コミットメッセージ（日本語）：** `02: EV/Sales倍率を17→15に修正、Exit再計算` のように **対象ファイル番号＋何をしたか**。
- **マージ前に pull/rebase** して衝突を解消。`.md` の衝突は両者の意図を残して手で解決。

---

## 5. レビュー規約

- 指摘には重大度プレフィックスを付ける：`Critical:` / `Important:` / `Nit:` / `FYI:`（無印＝必須）。
- **数字の指摘が最優先。** 採点軸③④は数字の整合で決まる。`［仮・要検証］` が残っていないか相互チェック。
- 「逃げずに数字で返せるか」をレビュー観点に（採点軸⑤QA対応力）。
